# onepool architecture

## Principles

1. **Session-based, not infrastructure.** A pool exists while its processes run. No
   daemons, no accounts, no state left behind (except checkpoints the user asked for).
2. **Heterogeneous-first.** Mixed OSes, GPU vendors, and speeds are the default case.
   Capability is measured (micro-benchmark on the real model), never assumed.
3. **WiFi is the network.** Every design choice assumes ~30–60 MB/s, lossy, shared.
   Anything that needs per-step gradient sync is out.
4. **One process, dynamic roles.** Every machine runs the same `onepool` process.
   The node that submits a job coordinates that job. No election, no Raft: if the
   coordinator dies, the job dies and workers return to the idle pool. That is a
   feature of a session tool, not a bug.

## Layers

| Package | Responsibility |
|---|---|
| `onepool.hw` | Hardware probe (torch + OS paths), readiness diagnosis, micro-benchmark (M3) |
| `onepool.discovery` | mDNS advertisement/browsing (`_onepool._tcp.local.`) + UDP broadcast fallback (M1) |
| `onepool.net` | TLS/TCP control plane, msgpack frames; tensor transfer for sync rounds (M1/M3) |
| `onepool.train` | DiLoCo trainer: inner AdamW loop, outer Nesterov sync, LoRA, sharding (M2/M3) |
| `onepool.jobs` | `run.yaml` job spec parsing, Python SDK (M2) |
| `onepool.dash` | FastAPI + WebSocket local dashboard (M1) |
| `onepool.cli` | Typer CLI: `up`, `join`, `train`, `doctor`, `status` |

## Training: why DiLoCo + LoRA

Naive data-parallel (DDP) synchronizes full gradients every step — 400MB/step for a
100M-param model in fp32. Dead on arrival over WiFi.

DiLoCo (DeepMind 2023; OpenDiLoCo, Prime Intellect 2024) restructures training:

- **Inner loop:** each worker runs H (~100) local AdamW steps on its own data shard.
- **Outer step:** workers all-reduce *pseudo-gradients* (θ_start − θ_local) and apply
  one Nesterov-SGD step to the shared weights. Communication drops 100–500×.
- **LoRA:** with a frozen (optionally 4-bit) base model, only adapter deltas sync —
  a 3B model with r=16 adapters is ~5–10MB int8-quantized per sync round. ~1s on WiFi.

Heterogeneity falls out naturally:

- Per-node batch size from measured VRAM + throughput.
- Workers need not run identical step counts between syncs; averaging is weighted by
  samples processed.
- A node missing the sync barrier (timeout 60s) is dropped, its shard redistributed.
  A joining node receives current weights and a shard at the next round.
- CPU-only nodes take small shards for tiny models, or serve as data/tokenization
  workers — every machine in the room contributes something.

## Security model

Discovery is open (anyone on the LAN sees that a pool exists), joining is not:
the session code shown by `onepool up` contains a shared secret. Control and data
connections run over TLS with a self-signed per-session certificate generated by the
first node; the secret authenticates peers. Threat model is "stranger on the same
coffee-shop WiFi", not nation states.

## Packaging

`torch` is deliberately not a hard dependency — the correct wheel depends on the
GPU vendor and OS. `onepool doctor` closes that gap: it detects hardware through OS
facilities (nvidia-smi, WMI, system_profiler, lspci) even when torch is absent and
prints the exact install command for this machine.
